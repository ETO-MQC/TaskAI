# SmartFocus——AI驱动的桌面任务管理应用·PRD

## 一、产品定位与核心理念

SmartFocus 是一款基于 **Tauri + React** 技术栈构建的跨平台桌面应用，核心理念是**「AI赋能任务管理」**——用户通过文字或语音对话框与AI对话，AI自动解析意图，完成任务的创建、分类、计时、追踪和统计，最大程度减少手动操作。市场上已经出现了类似 ArkClaw 通过AI对话指令完成「任务创建、进度追踪、节点提醒」全流程操作的产品先例，证明这一方向已被行业验证。同时，Todoist 的 Ramble 功能也已经验证了「语音输入→AI解析→结构化任务」的可行性，支持40多种语言的实时语音处理，自动提取日期、优先级等详情。SmartFocus 将在桌面端延续这一路径，并结合更丰富的可视化与模块互通能力，打造个人级别的AI任务中枢。


## 二、技术架构

### 2.1 整体架构

- **桌面框架**：Tauri 2.x（基于 Rust 后端），安装包体积 <5MB，内存占用 <30MB，启动速度约0.5秒，远优于传统 Electron 方案的80~200MB体积和150~300MB内存占用。
- **前端框架**：React 19 + TypeScript + Tailwind CSS，使用现代化组件库（如 shadcn/ui 或 Ant Design）确保界面丰富美观。
- **数据库**：SQLite（本地存储，无需联网即可使用），持久化任务、计时记录、用户配置等。参考「番茄工坊」等桌面应用的 SQlite 本地存储方案。
- **AI 集成**：集成 DeepSeek API（Chat Completions 接口），兼容 OpenAI SDK 格式，可通过 `https://api.deepseek.com/v1/chat/completions` 端点直接调用，提供多轮对话能力，支持 temperature 参数调节创造性。
- **语音支持**：集成浏览器内置的 Web Speech API（SpeechRecognition）进行语音识别，处理后发送给 DeepSeek API 进行意图解析，实现多语言语音输入支持。
- **架构模式**：前端通过 Tauri 的 `invoke` API 调用 Rust 后端命令，后端负责数据持久化、DeepSeek API 代理调用、系统通知等功能。

### 2.2 前后端交互流程

```
┌──────────────┐    invoke()     ┌──────────────┐
│  React 前端   │ ◄─────────────► │  Rust 后端    │
│  UI + 状态    │   JSON 数据     │  业务逻辑     │
└──────┬───────┘                 └──────┬───────┘
       │                                │
       │  Web Speech API / 文本输入     │  HTTP 请求
       ▼                                ▼
┌──────────────┐                 ┌──────────────┐
│  语音/文字    │                 │  DeepSeek    │
│  对话面板     │                 │  API         │
└──────────────┘                 └──────────────┘
                                        │
                               ┌───────┴────────┐
                               │  SQLite 数据库  │
                               │  本地持久化     │
                               └────────────────┘
```


## 三、功能模块详细设计

### 3.1 双视图布局

应用主界面采用**左侧导航栏 + 右侧内容区**的经典布局：

**左侧导航栏**：
- ① **任务**选项卡：进入任务管理主界面（To-Do List + 四象限视图）
- ② **计时**选项卡：进入计时/专注界面（番茄钟/正计时/倒计时）
- ③ **日程**选项卡：日历视图的日程安排表
- ④ **统计**选项卡：可视化数据仪表盘（进度圈、完成率、时间分布图表）
- ⑤ **AI助手**入口：右下角悬浮按钮或底部对话栏，全局可调用

**右侧内容区**：根据左侧选中的选项卡动态切换内容。

### 3.2 模块一：任务管理（To-Do List）

#### 核心功能清单

| 功能 | 描述 |
|------|------|
| 任务列表视图 | 支持按「未完成 / 已完成」分区展示，每条任务显示标题、描述、优先级标签、截止时间 |
| 四象限视图 | 基于艾森豪威尔矩阵（紧急/重要），将任务分为四象限：**重要且紧急、重要不紧急、紧急不重要、不重要不紧急** |
| 智能排序 | 支持按优先级、截止时间、创建时间、手动拖拽排序 |
| 任务详情 | 子任务拆解（无限层级）、备注（Markdown 支持）、标签分类、关联计时记录 |
| 快捷操作 | 勾选完成、左滑删除、长按拖拽调整优先级 |
| 日程关联 | 任务可关联具体日期和时间，自动同步到日程模块 |
| 提醒通知 | 系统级通知（通过 Tauri 的 notification API），支持到期提醒和提前提醒设置 |

#### AI 自动创建任务流程

```
用户输入（文字/语音） → 语音识别获取文本 → 发送给 DeepSeek API 进行意图解析
→ AI 提取结构化字段（任务标题、优先级、紧急度、截止时间、标签）
→ 若某字段不确定，AI 反问用户确认 → 确认后自动写入 SQLite → 前端实时刷新
```

需要从用户输入中提取的关键字段（以结构化 JSON 返回）：
- `title`：任务标题（必填）
- `description`：任务描述/备注
- `priority`：优先级（高/中/低）
- `urgency`：紧急度（紧急/不紧急）
- `importance`：重要性（重要/不重要）
- `quadrant`：自动归类到四象限（1-4）
- `deadline`：截止时间（ISO 日期格式）
- `estimated_duration`：预估耗时（分钟数）
- `tags`：标签数组
- `subtasks`：子任务数组
- `schedule_date`：计划执行日期

AI 在解析时需要遵循以下规则：
- 若用户输入不足以确定某字段，AI 生成反问提示，由用户确认后补充
- 支持**批量创建**：用户可一次性描述多个任务，AI 自动拆分成多个独立任务
- 支持**任务修改**：用户说「把刚才那个任务改成重要紧急」，AI 定位最近创建的任务并进行字段更新

### 3.3 模块二：计时/专注系统

#### 三种计时模式

**① 正计时模式（自由计时）** ：用户说「开始学计算机」，AI 自动识别任务主题并启动计时；学习完成后说「结束」，AI 自动停止、保存记录并关联到对应任务。

**② 番茄钟模式（25+5循环）** ：
- 标准循环：25分钟专注 + 5分钟休息，完成4轮后自动触发15~30分钟长休息
- 完成动画与提示弹窗，促进节律切换
- 支持自定义专注时长 / 休息时长 / 长休息时长 / 番茄轮数

**③ 倒计时模式**：用户设定固定时长，时间到后系统通知提醒。

#### 计时功能要点

- **可视化圆环形进度条**：实时显示剩余时间/已过时间，不同模式用不同配色（番茄钟→红色系、正计时→蓝色系、倒计时→橙色系）
- **最小化到系统托盘**：关闭主窗口后计时仍在后台运行，托盘图标显示缩略进度
- **暂停/继续/重置**：所有模式支持中途暂停和恢复
- **白噪音集成**（可选扩展）：内置雨声、咖啡馆等环境音，帮助沉浸专注
- **锁屏/免打扰联动**（可选扩展）：计时期间自动静音通知或锁定特定应用

#### 计时与任务的互通机制

这是 SmartFocus 的核心差异化设计。两个模块的数据通过以下方式实现深度互通：

1. **从任务启动计时**：在任务列表中，每条任务右侧有「开始专注」按钮，点击后自动跳转到计时模块并设定任务主题
2. **计时自动关联任务**：计时结束时，系统提示「这段时间做了什么？」，用户可选择关联已有任务或创建新任务
3. **计时记录自动回写**：完成计时后，自动在关联任务下生成时长记录（时间戳 + 时长 + 备注），用户可在任务详情页查看该任务累计投入时间
4. **AI 智能归因**：用户说「结束」，AI 根据对话上下文自动判断时段归属并关联到对应任务，无需手动指定
5. **预估 vs 实际对比**：任务完成时自动对比时长预估与实际耗时，生成完成报告

### 3.4 模块三：日程/日历视图

- **日历视图**：月视图/周视图/日视图三种切换
- **任务同步**：带日期的任务自动出现在对应日期格内
- **计时统计**：每日的计时总时⻓在日历格内可视化展示（颜色深浅表示时长多少）
- **拖拽调整**：支持在日历中拖拽任务调整日期
- **AI 日程规划**：用户说「帮我看看下周的安排」，AI 读取本地数据后生成日程总结
- **日程冲突检测**（可选扩展）：AI 自动检测同一天任务数量是否合理，主动建议调整

### 3.5 模块四：可视化统计仪表盘

#### 仪表盘组件

- **日环进度圈**（用户原始需求的核心可视化元素）：一个大圆环展示今日计划总时长 vs 实际完成时长，不同颜色段表示不同任务/类别占比
- **周统计卡片**：本周完成任务数、总专注时长、日均专注时长
- **四象限分布饼图**：展示各象限任务数量/时长占比
- **趋势折线图**：近7天/30天每日专注时长趋势
- **完成率仪表盘**：当日 / 本周 / 本月任务完成率百分比

统计数据来源于 SQLite 数据库中的任务表和计时记录表，通过 SQL 查询聚合后前端使用 ECharts 或 Recharts 进行可视化渲染。同时，参考四象限任务管理 App 的数据分析实践，长期趋势报告可帮助用户发现效率提升轨迹。


## 四、AI 助手核心能力设计

### 4.1 对话面板

- **全局入口**：右下角悬浮 AI 按钮或使用全局快捷键调出对话面板
- **输入方式**：文字输入框 + 麦克风按钮（点击开始录音，再点击结束）
- **对话历史**：保留最近30条对话记录，支持上下文理解
- **流式输出**：通过 DeepSeek API 的流式模式实时展示 AI 响应内容

### 4.2 AI 支持的操作指令集

| 操作类型 | 示例指令 |
|----------|---------|
| 创建任务 | 「帮我创建一个明天下午3点的会议任务，标记为重要紧急」 |
| 批量创建 | 「这周要做三件事：写周报、整理文件、约朋友吃饭」 |
| 修改任务 | 「把『写周报』改成重要不紧急」 |
| 删除任务 | 「删除『整理文件』这个任务」 |
| 查询任务 | 「我今天还有哪些未完成的任务？」 |
| 启动计时 | 「我开始写代码了」 |
| 结束计时 | 「我写完了」 |
| 查询进度 | 「今天一共专注了多久？」、「这周完成了多少个任务？」 |
| 获取建议 | 「帮我看看明天的时间安排合理吗？」 |

### 4.3 System Prompt 核心定义

AI 在接收到每条用户输入后，首先判断意图类型（`create_task` / `update_task` / `delete_task` / `query_task` / `start_timer` / `stop_timer` / `query_progress` / `get_advice` / `general_chat`），然后执行对应的结构化操作。对于解析不完整的字段，以 JSON 格式标记 `needs_clarification` 并生成反问文本。每条用户输入返回统一的 JSON 结构，包含 `intent`、`action`、`data`、`clarification`、`reply` 字段，前端根据返回结果执行对应的 UI 操作和刷新。


## 五、数据模型设计

### 5.1 核心数据表

**tasks 表**：存储所有任务，包含唯一 ID、标题、描述、优先级、紧急度、重要性、象限、状态、截止时间、预估时长、实际总时长、父级任务 ID、计划日期、标签和创建/更新时间。

**timer_records 表**：存储每次计时记录，包含唯一 ID、关联任务 ID（可空）、任务主题、计时模式、开始/结束时间、时长和备注。

**ai_conversations 表**：存储 AI 对话历史，包含唯一 ID、角色、内容和时间戳。

**user_settings 表**：存储用户配置，以键值对形式保存番茄时长、休息时长、长休息时长、番茄轮数、主题色、通知开关等参数。

### 5.2 数据互通关系

tasks ↔ timer_records 之间的关联通过 `task_id` 外键实现，这保证了以下互通场景：
- 查看某任务累计投入的所有计时记录
- 任务完成时自动汇总计时总时长并更新 `actual_total_duration`
- 统计模块按任务维度聚合计时数据生成报表

## 八、技术实现要点与注意事项

1. **DeepSeek API 调用**：通过 Tauri Rust 后端代理 API 请求（而非前端直接调用），以保护 API Key 安全。需实现指数退避重试机制以应对网络波动和 API 限流场景。

2. **语音识别**：使用浏览器内置 Web Speech API 的 SpeechRecognition 接口进行语音采集；若需更高精度，可升级为 DeepSeek 的语音模型或集成专门语音服务。

3. **本地数据安全**：所有数据默认本地存储，用户数据不出设备。如需未来扩展多端同步，可引入端到端加密的云同步方案。

4. **离线可用**：核心功能（任务 CRUD、计时、统计查询）完全不依赖网络。仅 AI 对话功能需要联网调用 DeepSeek API。

5. **系统通知**：通过 Tauri 的 notification 插件发送系统级通知，支持番茄钟结束提醒、任务到期提醒、日程提醒等功能。

6. **前端状态管理**：推荐使用 Zustand 或 Redux Toolkit 进行全局状态管理，管理任务列表、计时状态、AI对话历史等跨组件共享数据。

7. **动画性能**：圆形进度环优先使用 CSS transform + transition 实现，避免频繁触发重排影响计时精度，同时保证动画流畅度和低 CPU 占用。

二、AI System Prompt 完整定义
2.1 System Prompt 文本（供 Sprint 5 后端载入）
text
你是 SmartFocus 的智能任务与计时助手。你会收到用户的文字或语音输入，你的工作流程如下：

1. 判断用户的意图，从以下类别中选择一个：  
   create_task, update_task, delete_task, query_tasks, start_timer, stop_timer, query_progress, get_advice, general_chat

2. 根据意图提取结构化数据，并对缺失的必需字段通过反问用户补充。

3. 返回严格符合以下 JSON 格式的响应（不要包含任何多余的文本）：
{
  "intent": "create_task",
  "action": "create",
  "data": {
    "title": "任务标题",
    "description": "...",
    "priority": "high|medium|low",
    "urgency": "urgent|not_urgent",
    "importance": "important|not_important",
    "quadrant": 1,
    "deadline": "2026-05-06T15:00:00+08:00",
    "estimated_duration": 90,
    "tags": ["标签1"],
    "subtasks": [{"title":"子任务1"}],
    "planned_date": "2026-05-06"
  },
  "needs_clarification": false,
  "clarification": null,
  "reply": "已为你创建任务「任务标题」，放在了重要紧急象限，截止明天下午3点。"
}

字段填充规则：
- priority: 若用户未指定，默认 medium。
- urgency 和 importance: 根据描述推断，若提到时间紧迫则 urgency 为 urgent，否则 not_urgent；若提到重要则 importance 为 important，否则 not_important。
- quadrant: 按艾森豪威尔矩阵自动计算：
  1 = urgent & important
  2 = not_urgent & important
  3 = urgent & not_important
  4 = not_urgent & not_important
- deadline: 解析用户提到的时间，如“明天下午3点”→当前日期+1天15:00:00。未提供则留 null。
- planned_date: 若用户提到“今天做”“周三做”，解析为具体日期，否则留 null。
- estimated_duration: 若用户提到“大概要2小时”之类，转换为分钟数，否则 null。
- needs_clarification: 若 title 为空或无法推断，设为 true，并在 clarification 中输出反问问题。
- 支持批量创建：若用户说“帮我创建A任务和B任务”，则返回一个 JSON 数组，每个元素一个任务对象，intent 仍为 create_task。

对于 start_timer 意图：
- 提取主题（如用户说“开始学计算机”），主题存为 data.task_topic。
- reply 中提示“已开始计时「学计算机」”。

对于 stop_timer 意图：
- reply 中提示“计时已结束，本次专注X分钟”，data 中返回 duration 等。

其他规则：
- 始终保持友好简洁的中文回复。
- 不要输出除 JSON 以外的任何内容。
2.2 不确定时的交互范例
用户：“帮我创建任务”
→ AI：{"needs_clarification":true,"clarification":"请问任务的标题是什么？","reply":"..."}

四、数据模型补充说明
4.1 数据表字段详细约束
tasks.deadline 和 tasks.planned_date 使用 ISO 8601 字符串（例如 2026-05-06T15:00:00+08:00），前端统一使用 dayjs 解析与格式化。

tasks.quadrant 由后端根据 urgency 和 importance 自动计算，不允许前端直接写入，确保一致性。

tasks.actual_total_duration 单位为分钟，浮点数。

timer_records.mode 枚举：positive | pomodoro | countdown。

timer_records.task_id 可为 NULL（表示未关联任务）。

user_settings 表引入默认值逻辑：安装后首次启动自动插入默认设置（番茄25min，休息5min，目标每日8h，主题light）。

4.2 自动更新累计时长的 SQL 逻辑
stop_timer Rust 命令伪代码：

text
save timer_record (duration, task_id)
if task_id is not null:
   UPDATE tasks SET actual_total_duration = COALESCE(actual_total_duration, 0) + duration WHERE id = task_id
